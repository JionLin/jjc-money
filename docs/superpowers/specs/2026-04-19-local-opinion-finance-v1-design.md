# 本地观点 + 实时财报联动（轻量版 v1）设计

## 背景

当前仓库是一个按月份整理的 Markdown 内容归档，核心内容位于 `22-25year/`，同时新增内容也可能出现在 `26year/`。仓库中已有 `docs/indexes/` 作为检索辅助层，但索引不是唯一真相，最终结论仍需回到源文件核对。

目标是先落地一个轻量版问答流：当用户询问某只股票（如英伟达 / NVDA）时，能够联动本地历史观点与实时财报/估值数据，输出结构清晰、来源明确、避免混淆“历史观点”和“当前事实”的分析结果。

本版明确采用轻量方案：继续使用仓库现有索引与源文件工作流，不先做自定义 MCP、不做向量库、不做自动持久索引重建。这些作为后续待办保留。

## 目标

第 1 版需要支持：

1. 从 `docs/indexes/`、`22-25year/`、`26year/` 中检索与目标股票相关的历史观点。
2. 从外部数据源获取该股票的最新财报与估值数据。
3. 将历史观点与当前数据进行联动，但在输出中严格区分两类信息。
4. 形成一个稳定的输出结构，方便后续继续演进为自定义 MCP 或更强检索层。

## 非目标

本版不包含以下内容：

- 自定义 MCP 工具层
- 向量库 / 语义检索基础设施
- 自动持久索引构建或重建流程
- 对仓库内容做大规模格式重构
- 自动得出交易动作或下单能力

相关扩展已经作为后续任务单独记录。

## 适用范围

本地观点检索范围包括：

- `docs/indexes/`
- `22-25year/`
- `26year/`

其中默认优先使用索引缩小范围；如果索引未覆盖到 `26year/` 的新内容，允许直接在 `26year/` 源文件中做关键词检索并回读原文。

## 总体架构

轻量版 v1 采用三段式数据流：

1. **本地观点检索层**
   - 先查询 `docs/indexes/`，缩小到候选月份或候选文章
   - 再回到 `22-25year/` 或 `26year/` 的源文件提取原文观点
2. **实时数据获取层**
   - 获取最新财报、估值、行情相关指标
3. **融合输出层**
   - 把本地观点与实时数据分层呈现，再给出联动判断

该架构强调：

- 本地内容负责提供“观点框架”和历史语境
- 实时数据负责提供“当前校准”和事实更新
- 最终输出必须区分“原文观点”和“当前数据事实”

## 检索流程

当用户提出类似“英伟达如何”“NVDA 现在怎么看”这类问题时，处理流程固定为四步：

### 1. 本地检索

先查 `docs/indexes/`，然后查 `22-25year/` 与 `26year/` 源文件，定位与目标主题相关的文章或段落。

针对股票类问题，检索词通常包括：

- 股票代码，例如 `NVDA`
- 中文名称，例如 `英伟达`
- 相关扩展主题，例如 `估值`、`PE`、`Forward PE`、`仓位`、`买卖点`、`七巨头`

### 2. 原文提炼

从源文件中只提炼可被引用和验证的观点，例如：

- 当时的估值判断
- 仓位态度
- 风险提示
- 与时间背景强相关的结论

历史观点必须保留时间语境，不能被改写成“当前仍然成立”的断言。

提炼本地内容时还需要遵守以下规则：

- 默认按**从近到远**的时间顺序输出，优先让用户先看到最新观点，再看到更早观点
- 每条观点都要标明它来自**正文**还是**评论区**
- 如果观点出现在评论区，需要保留“读者提问 / 作者回复”这一层关系，避免把评论区回复误写成正文立场
- 在引用某条观点时，应补充该观点前后相邻的关键语句或简短上下文，帮助用户理解原话的连贯性，而不是只摘一小句孤立表述

### 3. 实时数据补充

查询外部数据源，获取当前关键指标，例如：

- 最新季度营收
- 毛利率 / 净利润 / 指引
- Forward PE
- 市值或行情快照
- 最近财报发布日期

### 4. 合并输出

最终输出先列本地原文观点，再列实时数据，再写综合结论。综合结论中必须明确区分：

- 哪些结论是当前可验证的事实
- 哪些判断是基于历史观点框架对当前数据的推导

## 数据源设计

### 官方事实来源

财报事实优先来自官方原始披露渠道：

1. **NVIDIA Investor Relations**
   - 财报公告
   - 股东信
   - 业绩演示材料
2. **SEC EDGAR**
   - 10-K
   - 10-Q
   - 8-K
3. 需要时可补充官方 earnings release PDF 或官方 presentation

这类来源主要用于：

- 最新季度营收
- 毛利率
- 净利润
- 管理层指引
- 财报发布日期
- 业务分部变化

### 聚合数据来源

聚合数据源用于补官方渠道不直接给出、但常见于投资问答的指标。

推荐优先级如下：

1. **Financial Modeling Prep (FMP)**
2. **Alpha Vantage**
3. **Finnhub**
4. **Yahoo Finance / yfinance**

这类来源主要用于：

- Forward PE
- DCF 或 DCF 现成结果
- 市值
- 估值倍数
- 分析师预期类字段
- 行情快照

## 降级策略

实时数据获取必须支持自动降级。

固定顺序为：

1. 先查官方来源获取财报事实
2. 再查 FMP 获取估值与市场指标
3. FMP 失败则降级到 Alpha Vantage
4. Alpha Vantage 失败则降级到 Finnhub
5. Finnhub 失败则降级到 Yahoo Finance / yfinance
6. 如果全部失败，则返回“本次仅能提供本地观点，实时数据缺失”

降级发生时，输出中必须明确说明已使用备用数据源，避免用户误以为所有指标都来自首选来源。

## 输出格式

每次回答固定为 4 段：

### 1. 结论摘要

用几句话回答“现在怎么看”。

其中来自本地档案的历史观点摘要，默认按**从近到远**的顺序组织，避免把较早观点放在最新观点之前。

### 2. 本地原文观点

列出来自 `22-25year/` 或 `26year/` 的关键观点，并带日期或来源定位。

输出顺序默认按**从近到远**排列。每条观点都应标明：

- 来源日期
- 来源文件
- 来自正文还是评论区
- 必要的前后文补充

如果引用的是评论区内容，应尽量连同对应的提问和作者回复关系一起呈现。

### 3. 实时财报 / 估值数据

列出最新财报指标与估值数据，并注明数据时间与来源。

### 4. 联动判断

明确拆成两类：

- **官网 / 数据事实**
- **基于历史观点框架的推导判断**

这样可以避免把“作者旧观点”和“当前事实”混为一体。

## 正确性与边界要求

### 事实边界

- 财报事实优先认官方来源与 SEC
- 市场估值类指标允许来自聚合数据源
- 每个关键数据点都应带来源标签

### 表达边界

- 不把旧文章中的观点表述为当前已验证结论
- 不把聚合站推导值表述为公司官网直接披露内容
- 不在数据缺失时伪造或猜测指标

### 仓库边界

- 继续沿用当前索引优先、源文件复核的仓库工作流
- 如果索引与源文件冲突，以源文件为准
- 若 `26year/` 尚未完全建立索引，允许直接查源文件补齐

## 后续演进方向

以下内容不在本版实现范围内，但已明确作为后续待办：

1. 自定义 MCP 层，用稳定接口封装本地观点检索与财报查询
2. 向量检索层，用于提升语义召回能力
3. 自动持久索引更新机制，覆盖 `22-25year/` 与 `26year/`

这些能力只有在轻量版证明工作流确实高频、有价值、且现有检索或响应效率成为瓶颈时再补充。
