# 美股财报 MCP 可用性验证设计

## 背景

当前仓库是一个以 Markdown 归档为主的内容仓库，不是应用代码库。本次工作不是直接搭建投研工作流，而是先验证 `获取美股财报信息mcp.md` 中提到的几类 MCP 候选是否真实存在、是否可安装，以及是否真能提供你关心的美股财报与估值数据。

这次验证采用混合方式：先做静态核验，再对 1-2 个最有希望的候选做本机轻量实测。这样可以在不一开始要求你提供 OAuth 或 API Key 的前提下，尽量给出有证据支撑的判断。

## 目标

本次验证需要完成以下目标：

1. 验证候选 MCP 是否有可靠来源，名称、包名、安装方式是否真实可落地。
2. 验证候选 MCP 是否能在本机进入可用路径，例如成功安装、被 Claude 识别、列出工具，或明确卡在凭证步骤。
3. 验证候选 MCP 的数据能力是否覆盖你关心的字段，而不是只停留在宣传描述。
4. 输出一份按证据分级的可用性结论，明确推荐下一步最值得接入的候选。

## 范围

本次验证范围限定为 `获取美股财报信息mcp.md` 中提到的以下候选：

- `Financial Datasets`
- `trading-mcp`
- `Alpha Vantage MCP`
- `Defeatbeta API`

重点核验的数据能力包括：

- Forward PE
- 财报三表
- Earnings
- Company Overview
- 基础估值指标
- 与 NVDA 相关的查询能力

本次不包含以下内容：

- 长期配置完整的生产级 MCP 工作流
- 代你完成交互式 OAuth 登录
- 代你申请或管理外部 API Key
- 基于返回数据直接做完整投研结论

## 验证策略

本次采用“静态核验 + 轻量实测”的混合方案。

原因是：

- 只做静态核验，无法证明候选真的能返回数据。
- 对全部候选做完整实测，成本高，而且很可能被 OAuth 或 API Key 阻断。
- 混合方案能先排除明显不靠谱的项，再把精力集中在最可能可用的候选上。

推荐执行顺序如下：

1. 先对全部 4 个候选做静态核验。
2. 再从中挑 1-2 个最有希望成功的候选做本机轻量实测。
3. 若遇到凭证门槛，则记录阻断点并暂停，不在本次阶段继续要求用户先补凭证。

## 验证流程

### 1. 静态核验

对每个候选检查：

- 是否能找到可靠项目主页、官方文档、npm 包或代码仓库
- 名称、包名、安装命令是否一致
- 是否仍在维护，或至少仍可找到有效安装入口
- 文档中宣称的工具名、能力名是否清晰可核

如果候选名称、包名、工具名互相对不上，默认将其降级为可疑项。

### 2. 安装前可行性检查

对每个候选补充检查：

- 是否依赖 Node / npx
- 是否需要 Claude MCP 配置
- 是否需要 OAuth
- 是否需要 API Key
- 是否存在额外环境变量或账户前提

这一步的目标是把“能跑到哪一步”提前讲清楚，而不是等失败后再回头梳理条件。

### 3. 本机轻量实测

从静态核验结果中挑选 1-2 个最可能成功的候选，尝试做到以下至少一项：

- 成功安装并被 Claude 识别
- 能列出工具
- 能发起 NVDA 相关查询并返回数据
- 明确卡在 OAuth / API Key / 账户权限步骤

本次轻量实测的重点是证明真实可用路径是否存在，而不是追求把所有候选都完整接通。

### 4. 字段映射核验

对于已找到可靠依据或已进入实测阶段的候选，检查其实际能力是否覆盖以下字段：

- Forward PE
- 三张财报表
- Earnings
- Company Overview
- 估值类指标

若文档只写“支持财务数据”但未能证明具体字段存在，则只能标为“能力范围待确认”，不能直接写成已支持。

### 5. 结论整理

所有结果统一整理成一张可比较的结论表，并附简洁推荐意见，避免把“文档存在”误写成“实际可用”。

## 证据标准

本次验证采用严格分级：

- **已验证**：来自本机命令输出、Claude MCP 实测结果，或官方文档的直接明确说明
- **可推定支持**：有较强文档依据，但未完成本机实测
- **待确认**：只看到了宣传描述，未见清晰工具或字段证明
- **可疑 / 不推荐**：名称、包名、命令或来源互相冲突，无法建立可靠对应关系

任何没有命令输出或明确官方佐证的能力，都不能写成“已支持”。

## 最终产出

最终输出包括三部分：

1. 一段简洁结论，总结哪些候选最靠谱。
2. 一张对比表，至少包含以下列：
   - 名称
   - 是否找到可靠来源
   - 安装命令是否有效
   - 是否需要凭证
   - 是否能列出工具
   - 是否能拿到 NVDA 相关数据
   - 是否覆盖 Forward PE / 三表 / Earnings / Overview
   - 当前结论
3. 一段下一步建议，明确最值得优先接入哪一个，以及原因。

## 停止条件

本次验证在满足以下条件后即可结束：

1. 4 个候选全部完成静态核验。
2. 至少 1-2 个候选完成本机轻量实测，或明确被凭证阻断。
3. 能给出“下一步先接哪个”的明确建议。
4. 如果需要你交互登录或补 API Key，则暂停并明确告知缺失项，不继续假装验证完成。

## 成功标准

若本次验证结束时，能够让你明确区分以下几类候选，就算成功：

- 已证实能走通基础路径的候选
- 可安装但被凭证阻断的候选
- 仅有文档说明、尚未证实的候选
- 缺少可靠依据、不建议继续投入的候选

这样你后续决定接哪个 MCP 时，就能基于证据，而不是基于宣传文案。
