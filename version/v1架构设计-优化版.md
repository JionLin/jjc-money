# jjc-money 智能数字分身 v1 架构设计（优化版）

**文档状态**: Draft v2
**文档目的**: 在 `v1-architecture-draft.md` 契约骨架的基础上，补齐工程落地肌肉，让架构既能通过评审，也能让工程师明天就动手写代码。
**核心目标**: 不是"会说金渐成口吻的话术机器人"，也不是"纯哲学的契约文档"，而是一个**以 Skill 为运行时合同、以 spring-ai-lab 为执行底座、可观测可测试可降级的问答系统**。

---

## 1. 问题定义

本项目的核心问题不是"能不能模仿一种语气"，而是：

> 当用户问"某个标的现在怎么看"时，系统能否同时做到：
> 1. 找到作者过往的真实历史观点
> 2. 获取当前最新事实
> 3. 按 Skill 规定的判断框架进行融合
> 4. 严格区分"历史观点"和"当前事实"

这决定了本系统本质上不是普通聊天应用，而是一个 **Skill 执行系统**。

## 3. V1 目标

V1 只聚焦最小闭环。

### 3.1 支持的问题类型

V1 仅支持两类：

1. **Historical View**：用户询问作者过去如何判断、如何操作
   - 例：`2025 年 4 月关税战时他是怎么操作的？`
2. **Ticker**：用户询问某个具体标的当前怎么看
   - 例：`NVDA 现在怎么看？能买吗？`

### 3.2 V1 成功标准（量化）

- ✅ Mode Router 在 Golden Set 上准确率 ≥ 90%
- ✅ Historical View 必须引用 `22-25year/` 或 `26year/` 原文（可验证文件路径）
- ✅ Ticker 必须包含 fresh data 时点标记，否则降级为"历史框架演练"
- ✅ 单次提问端到端成本 < $0.05，P95 延迟 < 30s
- ✅ 每次回答可在 trace 日志中追溯：mode → tool calls → 引用源 → fresh data → 最终输出

---

## 4. 非目标

V1 不做以下内容：

- 通用人格聊天机器人
- 多人格 / 多 Skill 系统
- Personal Portfolio Plan（即使用户主动引用持仓，V1 也降级到 Ticker 模式 + 提示"个人组合分析在 v2 支持"）
- 自动重建全文索引体系
- 向量检索（spring-ai-lab 已有 VectorSearchService，但 V1 不接入）
- 多租户 / 平台级 agent 编排
- Life Decision / 育儿 / 婚姻话题（直接拒绝并提示 v1 范围）

## 5. 核心设计原则

V1 必须服从 `jinjian-perspective` Skill 已定义的运行规则。

### 5.1 Freshness Gate

- 操作 / 估值类问题必须先获取实时数据
- 无法获取时只能标记"历史框架演练"
- 不允许在缺少实时事实时给出当前操作建议

### 5.2 Evidence Hierarchy

证据优先级：

1. 实时市场 / 财报 / filing 数据
2. 原始月度档案 (`22-25year/`, `26year/`)
3. 派生文档（topology、deep-analysis）
4. 索引或导航层

### 5.3 Source Verification

任何最终结论必须回到原始月度文件核验。派生文档只能帮助理解。

### 5.4 Advice Framing

输出框架、区间、条件和风险，不输出伪装成确定性的个性化指令。

### 5.5 Conflict Handling

当前事实与历史锚点冲突时，必须解释冲突来源，而不是强行把旧观点包装成当前结论。

## 7. Skill 加载与热更新策略

### 7.1 决策

V1 采用 **mtime 缓存 + 文件 watcher** 方案：

- 启动时加载一次到内存
- 每次请求前检查文件 `lastModified()`，若变化则重新读取
- `dev` profile 下强制每次请求读盘（无缓存）
- 失败 fallback：保留上一次成功加载的版本，并打印 ERROR 日志

### 7.2 工程实现

```java
@Component
public class SkillLoader {
    private static final Path SKILL_PATH = Paths.get(".agent/skills/jinjian-perspective/SKILL.md");
    private final AtomicReference<CachedSkill> cache = new AtomicReference<>();

    public String loadCurrent() { /* mtime 比对 + 重读 */ }
}

record CachedSkill(String content, long mtime, String version) {}
```

### 7.3 版本契约

- Skill 头部 `metadata.version` 必须可解析（如 `v2.4`）
- 系统启动日志必须打印加载到的 Skill 版本
- 当 Skill 版本与系统支持的最低版本不兼容时，启动失败而非静默降级

## 8. Mode Router 工程实现

### 8.1 收敛策略

V1 仅识别 2 种模式（Historical View / Ticker），其余降级处理：

| 模式 | V1 处理 |
|------|---------|
| Ticker | 走完整 SOP §7.1 |
| Historical View | 走 SOP §7.1 但跳过 Fresh Data 步骤，强制标"历史观点" |
| Personal Portfolio | 降级到 Ticker + 提示"个人组合在 v2 支持" |
| Portfolio | 降级到 Ticker + 提示"资产配置在 v2 支持" |
| Life Decision | 直接拒绝 + 提示"V1 仅支持投资类问题" |

### 8.2 路由实现：A + C 混合

**第一道：关键词正则白名单**（命中即定）
- Ticker 命中：`NVDA|MSFT|GOOGL|META|AMZN|AAPL|TSM|TSLA|BRK|...`（七巨头+台积电+常见美股）
- Historical 命中：`以前|当年|2024|2025|历史|曾经|当时`

**第二道：LLM 在主调用中自路由**（未命中走这里）
- 在 System Prompt 中嵌入 Mode Router 表
- 让模型在思考前先输出 `mode: Ticker` 标签
- 后端从流式输出第一个事件中拿到 mode，作为后续路由依据

### 8.3 路由结果显式化

```text
event: mode-detected
data: {"mode": "Ticker", "ticker": "NVDA", "confidence": "high", "via": "regex"}
```

## 9. 检索策略与 Tool Calling

### 9.1 决策：使用 Spring AI Tool Calling

理由：
- Skill SOP §7.1 本身是多步推理（Step 1-8），让模型驱动检索符合 SOP 精神
- 后端预拼上下文会过早收敛证据，损失灵活性
- spring-ai-lab 已有 `WeatherTools` 模板，工程成本低

### 9.2 V1 工具清单

| Tool 名 | 输入 | 输出 | 用途 |
|---------|------|------|------|
| `searchLocalArchive` | `keyword: String, maxFiles: Int=5` | List<{file, sectionTitle, excerpt}> | 在 `22-25year/` `26year/` 检索原文 |
| `readArchiveSection` | `filePath: String, anchor: String` | full section text | 读取命中片段的完整上下文 |
| `getFreshTickerData` | `ticker: String` | {price, fwdPE, eps, fcf, asOfDate, source} | 获取实时价 + 估值指标 |
| `searchDerivedDocs` | `keyword: String` | List<{file, summary}> | 检索派生分析（topology / deep-analysis） |

### 9.3 检索路径（与 SKILL.md §6 一致）

```text
overview (archive-index.md)
    ↓
raw source search (22-25year/, 26year/)
    ↓
verify in raw section
    ↓
optionally enrich with derived docs
```

### 9.4 关键词优先级

针对标的类问题，工具内部默认搜索：股票代码 → 中文名 / 英文名 → 估值 / 锚点 / 加仓 / 减仓 / 负成本 / 仓位 / VIX / 现金流。

## 10. 实时数据接入工程规范

### 10.1 数据源优先级链

```
Priority 1: financial-datasets MCP（如已配置）
    ↓ 不可用 / 失败
Priority 2: 公司 IR 页面（针对深度报告问题）
    ↓ 不可用 / 失败
Priority 3: web search（兜底）
    ↓ 不可用 / 失败
Priority 4: user-provided（标注 unverified）
    ↓ 全部失败
Fallback: 标记"历史框架演练"，禁止输出当前操作建议
```

### 10.2 缓存策略

| 数据类型 | TTL | 存储 |
|---------|-----|------|
| 实时价格 | 5 分钟 | Redis |
| Forward PE / EPS / FCF | 1 小时 | Redis |
| 财报 / filings | 24 小时 | Redis |
| VIX / 利率 | 15 分钟 | Redis |

### 10.3 失败降级

- 单次工具调用失败 → 重试 1 次（指数退避）
- 重试仍失败 → 在 trace 中记录 `tool.failed`，继续后续步骤但跳过该数据点
- 关键数据全部失败 → 触发 §5.1 Freshness Gate，输出"历史框架演练"

## 11. 输出契约（外壳 + 内容嵌套）

### 11.1 双层契约关系

```
┌─────────────────────────────────────────────────────────┐
│ 系统级回答外壳（本文 §11.2）                            │
│  ┌───────────────────────────────────────────────────┐ │
│  │ Skill Output Contract（SKILL.md §5.2）            │ │
│  │   A: Ticker Analysis                              │ │
│  │   B: Portfolio / Asset Allocation（V1 不启用）    │ │
│  │   C: Personal Portfolio（V1 不启用）              │ │
│  └───────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 11.2 V1 系统级回答外壳（必须 4 段齐全）

1. **问题归类与前提**
   - 当前问题属于 Historical View / Ticker
   - 是否具备 fresh data
   - 是否存在关键前提缺失

2. **历史观点证据**
   - 时间 / 来源文件路径 / 正文 or 评论区 / 原文上下文

3. **当前事实**
   - 数据时点 / 主要指标 / 来源标签

4. **联动判断**
   - 沿用历史框架的当前结论
   - 最大风险与反方观点
   - 哪些是推导，不是历史原话

### 11.3 Ticker 模式下，第 4 段直接复用 SKILL.md Output Contract A 的内部字段（趋势 / 情绪 / 估值快扫 / 操作框架 / 风险与反方 / 免责声明）。

## 12. 流式 SSE 事件协议

为让前端可以渐进式展示多步推理，定义以下事件类型：

```text
event: trace-start          data: {"traceId": "..."}
event: mode-detected        data: {"mode": "Ticker", "ticker": "NVDA", "via": "regex"}
event: tool-call            data: {"tool": "searchLocalArchive", "args": {...}}
event: source-cited         data: {"file": "26year/26-04月.md", "section": "..."}
event: fresh-data           data: {"asOf": "2026-04-25", "price": 165.3, "source": "financial-datasets"}
event: answer-chunk         data: "..."   # 主体内容流式片段
event: disclaimer           data: "以上仅为个人看法..."
event: trace-end            data: {"costUsd": 0.032, "latencyMs": 18420}
```

前端按事件类型渲染：模式徽章、工具调用列表、引用卡片、流式答案、免责声明、底部成本与延迟。

## 13. 部署形态与前端要素

### 13.1 部署形态

- **单 jar 部署**：前端 `npm run build` 产物拷贝到 `spring-ai-lab/src/main/resources/static/`
- **零跨域**：前后端同源，无需 Nginx
- **运行**：`java -jar spring-ai-lab.jar` 一行起服务

### 13.2 前端必备 UI 要素

| 要素 | 说明 |
|------|------|
| 消息流 | 流式 token 展示，支持 Markdown + 代码高亮 |
| 模式徽章 | 顶部显示当前问题被路由到的模式（Ticker / Historical View） |
| 数据时点 | Fresh data 显示为可点击徽章，hover 显示来源 |
| 来源链接 | 引用的 `22-25year/` `26year/` 文件路径，支持点击跳转或展开摘录 |
| 工具调用面板 | 可折叠，展示本次回答用了哪些工具 |
| 免责声明 | 固定在每条 ticker 类回答底部 |
| 成本显示 | 调试用，可通过 `?debug=1` 开启 |

### 13.3 技术栈

- Vue 3 + Vite（保持原版选型）
- 不引入复杂状态管理，单页聊天足够
- SSE 客户端用原生 `EventSource`

## 14. 可观测性

### 14.1 必备追溯字段

每次请求产生一条结构化 trace，包含：

```json
{
  "traceId": "...",
  "userId": "anonymous",
  "question": "...",
  "skillVersion": "v2.4",
  "mode": "Ticker",
  "ticker": "NVDA",
  "toolCalls": [
    {"tool": "searchLocalArchive", "args": {...}, "latencyMs": 120, "success": true},
    {"tool": "getFreshTickerData", "args": {...}, "latencyMs": 850, "success": true}
  ],
  "citedSources": [
    {"file": "26year/26-04月.md", "anchor": "NVDA-2026-04-15"}
  ],
  "freshData": {"asOf": "2026-04-25", "source": "financial-datasets"},
  "outputLength": 1820,
  "costUsd": 0.032,
  "latencyMs": 18420,
  "status": "ok"
}
```

### 14.2 日志输出

- 复用 `TraceIdFilter` 注入 traceId 到 MDC
- 每个工具调用打印 INFO 日志
- 失败 / 降级路径打印 WARN 日志
- 启动时打印 Skill 版本与加载路径

### 14.3 调试入口

- `GET /debug/trace/{traceId}` 返回完整 trace（仅 dev profile）
- `GET /debug/skill` 返回当前加载的 Skill 内容与 mtime（仅 dev profile）

| 2 | 微软最近跌了不少，是机会吗？ | Ticker | Forward PE 区间 + 加仓区参考 + 风险反方 | 确定性买入指令 |
| 3 | 2025-04 关税战时他是怎么操作的？ | Historical View | 引用原文（带文件路径）+ "历史观点"标注 | 当前操作建议 |

| 6 | 我手里 NVDA 仓位 50% 太高了吗？ | 降级 | "V1 不支持个人组合，已降级为通用判断"提示 + Ticker 框架 | 个性化下单建议 |
| 7 | 现在该如何配置防守型账户？ | 降级（Portfolio） | "V1 不支持资产配置详细方案"提示 | 详细 BRK / TLT / SCHD 比例 |
| 8 | 我应该辞职吗？ | 拒绝 | "V1 仅支持投资类问题"提示 | 任何人生建议 |
| 9 | A 股茅台还能拿吗？ | Ticker → Avoid 或能力圈外 | "粪坑检测"或"能力圈外"声明 | 茅台估值深度分析 |
| 10 | UNH 能抄底吗？ | Ticker | 粪坑检测 4 项 + 历史引用（如果作者有提及） | 拍脑袋的 yes/no |

每次 Skill 升级或代码改动后，需重跑 Golden Set 并归档结果。

## 16. 失败模式与降级矩阵

| 失败场景 | 检测方式 | 降级动作 | 用户感知 |
|---------|---------|---------|---------|
| Skill 文件读取失败 | mtime check 异常 | 使用上次缓存版本 | 顶部提示"使用旧版 Skill" |
| Skill 版本不兼容 | 启动时检查 | 启动失败 | 服务不可用，运维介入 |
| Mode Router 无法识别 | 正则未命中 + LLM 输出非法 | 默认走 Ticker | 模式徽章显示"Ticker (兜底)" |
| 工具调用失败 | 异常 / timeout | 重试 1 次 → 跳过 | 来源面板显示"部分数据缺失" |
| Fresh data 全部失败 | 所有 Provider 失败 | Freshness Gate 触发 | 输出"历史框架演练"标签 |
| LLM 引用了不存在的文件 | 后端校验文件路径 | 在 trace 标 hallucination | 引用卡片标"未验证" |
| 用户问 V1 非目标问题 | Mode 识别为 Life / Portfolio | 降级或拒绝 | 提示 v2 范围 |
| Token 超出预算 | 累计成本检查 | 拒绝继续生成 | 提示"超出本次预算" |

## 17. 成本与性能预算

### 17.1 单次提问预算

| 指标 | 目标 | 上限 |
|------|------|------|
| LLM 调用次数 | ≤ 2 次 | 4 次 |
| Token 总量 | ≤ 8K | 16K |
| 工具调用次数 | ≤ 3 次 | 6 次 |
| 端到端延迟 P50 | ≤ 12s | 20s |
| 端到端延迟 P95 | ≤ 25s | 30s |
| 单次成本 | ≤ $0.03 | $0.05 |

### 17.2 超限处理

- 超过上限时拒绝继续生成，提示用户"问题过于复杂，请拆分"
- 超过目标但未到上限时，trace 标 `cost.warning`
