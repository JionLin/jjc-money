# jjc-money 智能数字分身 v1 架构设计（终稿候选）

**文档状态**: Draft v3  
**文档目的**: 在 `v1-architecture-draft.md` 的契约骨架与 `v1架构设计-优化版.md` 的工程化方案基础上，收敛出一版既贴近当前项目现实、又足够指导实现的 v1 架构文档。  
**核心目标**: 本系统不是“会说金渐成口吻的话术机器人”，而是一个 **以 Skill 为运行时合同、以 spring-ai-lab 为执行底座、以原始历史语料为证据、以实时事实为校准、可观测可测试可降级的问答系统**。

---

## 1. 问题定义

本项目的核心问题不是“能不能模仿一种语气”，而是：

> 当用户问“某个标的现在怎么看”时，系统能否同时做到：
> 1. 找到作者过往的真实历史观点
> 2. 获取当前最新事实
> 3. 按 Skill 规定的判断框架进行融合
> 4. 严格区分“历史观点”和“当前事实”

这决定了本系统本质上不是普通聊天应用，而是一个 **Skill 执行系统**。

## 3. V1 目标

V1 只聚焦最小闭环。

### 3.1 支持的问题类型

V1 仅支持两类问题：

1. **Historical View**
   - 用户询问作者过去如何判断、如何操作
   - 例：`2025 年 4 月关税战时他是怎么操作的？`

2. **Ticker**
   - 用户询问某个具体标的当前怎么看
   - 例：`NVDA 现在怎么看？能买吗？`

### 3.2 成功标准

- 能正确执行 Skill 的判断流程
- 能回到本地原始语料核对历史观点
- 能接入最新事实数据
- 能明确区分历史观点、当前事实、以及基于框架的联动判断
- 每次回答都能在日志 / trace 中回溯：mode、tool calls、引用源、fresh data、最终输出

### 3.3 目标指标

| 指标 | 目标 | 上限 |
| --- | --- | --- |
| 单次成本 | ≤ $0.03 | $0.05 |
| 端到端延迟 P50 | ≤ 12s | 20s |
| 端到端延迟 P95 | ≤ 25s | 30s |
| LLM 调用次数 | ≤ 2 次 | 4 次 |
| 工具调用次数 | ≤ 3 次 | 6 次 |

## 4. 非目标

以下内容不属于 V1 范围：

- 通用人格聊天机器人
- 多人格 / 多 Skill 系统
- 自动重建全文索引体系
- 向量检索优先方案
- 复杂个人组合管理
- 多租户产品化
- 平台级 agent 编排系统
- Life Decision / 育儿 / 婚姻等非投资问答

说明：

- 用户若询问 `Portfolio` / `Personal Portfolio`，V1 统一降级处理，不给出个性化组合处方
- 用户若询问非投资话题，V1 直接拒绝并提示当前范围

## 5. 核心设计原则

V1 必须服从 `jinjian-perspective` Skill 已定义的运行规则。

### 5.1 Freshness Gate

- 操作类 / 估值类问题必须先获取实时数据
- 无法获取实时数据时，只能输出“历史框架演练”
- 不允许在缺少实时事实时给出当前操作建议

### 5.2 Evidence Hierarchy

证据优先级如下：

1. 实时市场 / 财报 / filing 数据
2. 原始月度档案文件（`22-25year/`, `26year/`）
3. 派生文档（topology、deep-analysis 等）
4. 索引或导航层

### 5.3 Source Verification

任何最终结论都必须回到原始月度文件核验。  
派生文档只能帮助理解，不可替代原文。

### 5.4 Advice Framing

输出的是框架、区间、条件和风险，不是伪装成确定性的个性化指令。

### 5.5 Conflict Handling

如果当前事实与历史锚点冲突，必须解释冲突来源，而不是强行把旧观点包装成当前结论。

### 5.6 Progressive Degradation

系统遇到不完整数据、无法识别模式、工具失败时，应优先降级，而不是硬答。

## 7. Skill 加载与热更新策略

### 7.1 V1 决策

V1 采用 **mtime 缓存检查** 方案，不引入文件 watcher：

- 启动时加载一次 Skill 到内存
- 每次请求前检查 Skill 文件的 `lastModified()`
- 若文件变化，则重新读取并替换缓存
- `dev` profile 下允许强制每次请求读盘
- 若重读失败，则保留上一次成功加载的版本，并记录 ERROR 日志

### 7.2 这样设计的原因

- 足够满足 prompt / Skill 调试效率
- 明显简单于 watcher 方案
- 更贴合当前仓库的单机开发形态
- 出问题时更容易调试和回滚

### 7.3 版本契约

- Skill 头部 `metadata.version` 必须可解析
- 服务启动时必须打印当前加载的 Skill 版本
- 若 Skill 版本低于系统要求的最小兼容版本，则启动失败而非静默降级

## 8. Mode Router 工程实现

### 8.1 V1 收敛策略

V1 只正式支持两种模式：

| 模式 | V1 处理 |
| --- | --- |
| `Ticker` | 走完整 Ticker SOP |
| `Historical View` | 走 Historical SOP，跳过 fresh data 依赖 |
| `Portfolio` | 降级到通用 Ticker 说明，不给个性化组合建议 |
| `Personal Portfolio` | 降级到通用 Ticker 说明，不给个性化组合建议 |
| `Life Decision` | 直接拒绝并提示当前范围 |

### 8.2 路由方式

V1 采用 **两阶段路由**：

1. **第一阶段：规则路由**
   - 关键词 / 正则直接命中
   - 例如：`NVDA|MSFT|GOOGL|META|AMZN|AAPL|TSM|TSLA|BRK`
   - 历史问题特征词：`以前|当年|2024|2025|历史|曾经|当时`

2. **第二阶段：独立分类调用**
   - 对第一阶段未命中的输入，单独触发一次轻量分类判断
   - 输出结构化结果，如 `{mode: "Ticker", confidence: "medium"}`
   - 分类结果与最终回答生成分离，避免从正文流里反解析 mode

### 8.3 不采用“正文前先吐 mode 标签”的原因

- 这种方式会把控制协议嵌进自然语言流
- 前端与后端协议会更脆弱
- 一旦模型输出格式偏移，整个路由链就会抖动

因此，V1 路由结果必须在回答正文开始前独立确定。

## 9. 检索策略与 Tool Calling

### 9.1 决策

V1 使用 Spring AI Tool Calling，让模型在 Skill SOP 约束下按需调用检索工具。

理由：

- Skill 的 Ticker / Historical SOP 本身就是多步推理
- 模型驱动检索更贴近 Skill 精神
- `spring-ai-lab` 已有 `WeatherTools` 模板，工程成本低

### 9.2 V1 工具清单

| Tool 名 | 输入 | 输出 | 用途 |
| --- | --- | --- | --- |
| `searchLocalArchive` | `keyword`, `maxFiles` | `[{file, anchor, excerpt}]` | 检索 `22-25year/` / `26year/` 原文 |
| `readArchiveSection` | `filePath`, `anchor` | 完整片段文本 | 读取命中上下文 |
| `searchDerivedDocs` | `keyword` | `[{file, summary}]` | 检索派生分析文档 |
| `getFreshTickerData` | `ticker` | `{price, fwdPE, eps, fcf, asOfDate, source}` | 获取实时事实 |

### 9.3 推荐检索路径

```text
archive overview (optional)
        ↓
raw source search (22-25year/, 26year/)
        ↓
open matched section
        ↓
verify historical claim
        ↓
optionally enrich with derived docs
```

### 9.4 关键词优先级

针对标的类问题，优先搜索：

- 股票代码
- 中文名 / 英文名
- 估值
- 锚点
- 加仓
- 减仓
- 负成本
- 仓位
- VIX
- 现金流

## 10. docs/indexes 的定位

### 10.1 V1 定位

`docs/indexes` 不是核心资产，而是可替换的导航层。

### 10.2 V1 建议

- `archive-index.md`：可保留为月份级 overview
- `monthly/YYYY-MM.md`：不再作为必经层

### 10.3 原因

- 真相始终在原始月度文件中
- 当前语料规模仍允许直接搜索原文
- monthly indexes 的维护成本较高，且容易过期
- Skill 真正依赖的是“缩小范围 + 回原文核验”，而不是必须经过 article-level 索引

### 10.4 结论

V1 推荐默认工作流为：

```text
overview + raw source
```

而不是：

```text
overview + monthly indexes + raw source
```

## 11. 实时数据接入工程规范

### 11.1 数据源优先级

V1 采用分层降级：

```text
Priority 1: 已配置的结构化数据源
    ↓
Priority 2: 公司 IR / filings
    ↓
Priority 3: web search
    ↓
Priority 4: user-provided (标记 unverified)
    ↓
Fallback: 历史框架演练
```

说明：

- 这里不把某个具体 MCP 名称写死为唯一依赖
- 若未来使用 `financial-datasets MCP`，它属于 Priority 1 的一种实现，而不是文档唯一前提

### 11.2 缓存策略

| 数据类型 | TTL | 存储 |
| --- | --- | --- |
| 实时价格 | 5 分钟 | Redis |
| Forward PE / EPS / FCF | 1 小时 | Redis |
| 财报 / filings | 24 小时 | Redis |
| VIX / 利率 | 15 分钟 | Redis |

### 11.3 失败降级

- 单次关键调用失败：重试 1 次
- 重试仍失败：记录 `tool.failed`
- 关键事实全部失败：触发 Freshness Gate，输出“历史框架演练”

## 12. 输出契约

### 12.1 双层契约

V1 输出由两层组成：

1. **系统级回答外壳**
2. **Skill 内部输出合同**

系统负责把 Skill 结果放进统一可展示、可追溯、可降级的壳中。

### 12.2 系统级回答外壳

V1 回答统一分为四段：

1. **问题归类与前提**
   - 当前问题属于 `Historical View` / `Ticker`
   - 是否具备 fresh data
   - 是否存在关键前提缺失

2. **历史观点证据**
   - 时间
   - 来源文件路径
   - 正文 / 评论区 / 作者回复
   - 原文上下文

3. **当前事实**
   - 数据时点
   - 主要指标
   - 来源标签

4. **联动判断**
   - 沿用历史框架的当前结论
   - 最大风险与反方观点
   - 哪些是推导，不是历史原话

### 12.3 Ticker 模式内容

在 `Ticker` 模式下，第 4 段内部可复用 Skill 的 Ticker Output Contract 字段，如：

- 趋势判断
- 情绪温度
- 估值快扫
- 操作框架
- 风险与反方
- 免责声明

## 13. 流式输出协议

### 13.1 V1 现实约束

当前后端流式接口采用 `POST /chat/stream` 语义，因此前端 **不能直接使用原生 `EventSource`**。

V1 推荐继续采用 **`fetch` + ReadableStream** 风格消费流式响应，保持与现有后端接口形态一致。

### 13.2 为什么不选原生 EventSource

- 原生 `EventSource` 只支持 `GET`
- 当前流式接口需要提交 JSON body
- 若为了 `EventSource` 改接口，会增加额外协议调整成本

### 13.3 逻辑事件类型

即便底层继续使用 `fetch` 流，系统仍可在流内容上组织这些逻辑事件：

```text
trace-start
mode-detected
tool-call
source-cited
fresh-data
answer-chunk
disclaimer
trace-end
```

前端可按逻辑事件渲染模式徽章、工具调用面板、来源卡片、回答正文、免责声明与调试信息。

## 15. 可观测性

### 15.1 每次请求需可追溯

每次请求至少记录：

- `traceId`
- 用户问题
- Skill 版本
- mode
- ticker（如有）
- tool calls
- cited sources
- fresh data 来源与时点
- cost
- latency
- degrade status

### 15.2 日志规则

- 复用 `TraceIdFilter` 注入 traceId
- 每个工具调用打印 INFO 日志
- 失败 / 降级路径打印 WARN 日志
- Skill 加载、版本变化、重载失败打印启动 / 运行日志

### 15.3 调试接口

开发环境可保留：

- `GET /debug/trace/{traceId}`
- `GET /debug/skill`

仅供 dev profile 使用。

## 17. 失败模式与降级矩阵

| 失败场景 | 检测方式 | 降级动作 | 用户感知 |
| --- | --- | --- | --- |
| Skill 文件读取失败 | mtime 检查异常 | 使用上次缓存版本 | 顶部提示使用旧版 Skill |
| Skill 版本不兼容 | 启动检查 | 启动失败 | 服务不可用 |
| Mode Router 无法识别 | 规则未命中 + 分类失败 | 默认走 `Ticker` 兜底 | 模式标记为兜底 |
| 工具调用失败 | 异常 / timeout | 重试 1 次后跳过 | 提示部分数据缺失 |
| Fresh data 全部失败 | 所有 Provider 失败 | 输出历史框架演练 | 显式标注降级 |
| LLM 引用了不存在文件 | 后端路径校验 | 引用标未验证 | trace 记录异常 |
| 用户问 V1 非目标问题 | 模式识别 | 降级或拒绝 | 明确提示范围 |
| Token / 成本超限 | 成本累计检查 | 停止继续生成 | 提示拆分问题 |

## 18. 演进路线

### Phase A · V1.0
